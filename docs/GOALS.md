# RDB 项目目标

## 项目概述

rdb-mhc (Resource Database - MetaHub & TemplateClient) 是一个面向**团队协作**的数据资产管理系统，使用 FastAPI + PostgreSQL + Svelte 技术栈构建。

**核心理念**：RDB 本质上是"带标签的文档管理系统"，存储约定好结构的轻量可序列化数据，为散落的资源建立统一入口，通过多维度标签实现灵活分类，支持跨项目检索与交叉分析。

## 系统架构

```
┌─────────────────────────────────────────────────────┐
│                    rdb-mhc                          │
├──────────────────────┬──────────────────────────────┤
│      MetaHub         │     TemplateClient           │
│  ────────────────    │  ─────────────────────────   │
│  • 标签系统管理      │  • 数据存储与版本控制         │
│  • 层级化标签分类    │  • 多种数据类型的可视化       │
│  • 元数据管理        │  • 用户交互界面              │
│  • 资源元信息关联    │  • 数据的增删改查             │
└──────────────────────┴──────────────────────────────┘
```

## 本质分析

RDB 只存储约定好结构的数据。不同的数据类型类似于每种数据是一个文件类型，有不同的程序打开。

**结构化的代价**：为了实现数据结构化，带来的是表达能力的限制，RDB 需要使用者了解数据的上下文环境。文档通常需要将各种信息整合，把来龙去脉解释清楚；而 RDB 只存储核心的结构化数据。

**与文档的关系**：RDB 不能替代文档。多数情况下我们发布的是文档，包含数据的信息，但是经过渲染后的图片或表格。RDB 的目标是发布原始数据本身，可以直接被下游程序使用。

## 核心功能

### 1. 标签系统 (MetaHub)

MetaHub 提供**多维度层级化标签系统**：

- **层级化标签**：支持任意深度的父子标签关系
- **多维度分类**：一个资源可从多个维度打标签
  - 项目维度：如「项目A」
  - 测试维度：如「测试批次-001」
  - 设备维度：如「设备型号-X」
  - 特性维度：如「产品特性-a/b/c」
- **层级搜索**：搜索父标签时自动包含子标签内容
- **多标签分类**：资源在同一分类下可有多个标签（非单一树状路径）

### 2. 数据资产管理 (TemplateClient)

TemplateClient 是数据的统一存储与展示层：

- **RDB 存储**：存储各类轻量可序列化数据
- **多数据类型可视化**：针对已知数据类型，提供交互式可视化界面
  - 数值表格：可排序、筛选、导出
  - Mermaid 图表：渲染预览
  - 图片结构：网格/画廊视图
  - JSON 编辑器：树状/源码视图
- **资源模板**：不同类型数据有不同的快速添加界面，一键创建标准化资源
- **聚合分析**：支持同类数据的批量获取与趋势可视化
- **跨项目对比**：汇集同类型数据，支持跨项目数据对比

### 3. 资源模块设计

模块、资源类型、数据类型是同样的概念。同一种资源类型可以定义不同的模块，把上下文环境限定在某个范围内。

**基础模块**：

- **Generic**：宽泛的信息表达，不设置特定的表单结构，可使用 Markdown
- **Mermaid 流程图**：记录产品变化路径、产品结构分解等
- **数据表格**：经典行列数据，支持交叉对比
- **图片集**：图片上传加备注，作为典型特征目录

**定制模块**：

- **产品信息**：记录产品主要参数
- **统计性数据**：跨项目对比时的难点是"域"不一定相同，需要设计合理的处理方案
- **领域专用**：振动噪声、产品可运行空间、运行参数定义等

### 4. 数据交换

在模块的前端编辑呈现、前后端沟通、后端提供 API 服务中，最重要的是数据结构与数据交换。

- **实际存储**：最终持久化数据在数据库中的形式（JSON 字符串、原始文本、二进制结构数据）
- **后端对象**：后端能够将持久化数据还原为对象，向外只显示稳定的、简单的结构
- **传输结构**：通过 JSON 为前端或其他客户端提供结构化数据，用 Swagger 编写 API 文档
- **前端展示**：将信息渲染展示出来

### 5. 版本控制

- 记录资源变更历史
- 支持查看差异（diff）

### 6. API 接口

- 提供 RESTful API
- 支持分析程序推送结果到 RDB
- 可被其他应用复用 TC 的数据管理能力

### 7. 权限管理

- **基于角色的访问控制** (RBAC)
- 支持**私密资源**：可创建仅创建者可见的私有资源
- 团队成员可访问团队共享资源

## 设计原则

- **统一入口**：所有资源集中管理，快速检索
- **多维度分类**：标签系统支持非树状的灵活分类
- **层级搜索**：搜索父标签自动包含子标签
- **可复用**：TC 可作为其他应用的数据层，复用增删改查逻辑
- **易用性优先**：优先保证用户体验和操作便捷性
- **数据导向**：发布原始数据而非渲染结果，供下游直接使用

## 典型使用场景

### 场景：硬件测试结果管理

**标签定义（MetaHub）：**
```
项目
├── SmartHome (父)
│   ├── SmartLight
│   └── SmartSensor
└── Wearable
    └── SmartWatch

设备型号
├── D-100
└── D-200

测试环境
├── 室内-常温
└── 户外-低温

特性
├── 功耗
├── 连接稳定性
└── 响应延迟
```

**数据存储（TemplateClient）：**
```
资源1: 测试记录-001
├── 类型: 性能数据表
├── 内容: { latency: 120ms, power: 2.3W, ... }
└── 标签: [SmartHome/SmartLight, D-100, 室内-常温, 响应延迟]

资源2: 测试记录-002  
├── 类型: Mermaid时序图
├── 内容: { mermaid: "sequenceDiagram..." }
└── 标签: [SmartHome/SmartSensor, D-200, 户外-低温, 连接稳定性]
```

**检索与可视化：**
- 搜索"室内-常温 + 响应延迟" → 找到所有相关测试，对比结果
- 聚合所有"功耗"数据 → 生成趋势图表
- 通过 API 推送新测试结果 → 自动关联标签

## 技术选型

- **后端**：FastAPI + asyncpg (Python)
- **数据库**：PostgreSQL
- **前端**：Svelte 5 + TypeScript
